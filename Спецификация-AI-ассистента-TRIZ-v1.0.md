## 1. Product Vision

### Проблема пользователя

Инженеры и аналитики R&D тратят часы на поиск релевантных TRIZ-приёмов, разбор противоречий и формулировку гипотез — знания размазаны по учебникам, внутренним отчётам и опыту отдельных экспертов. Без структурированного методологического сопровождения решения получаются интуитивными, слабо воспроизводимыми и плохо документируемыми для аудита и передачи в команду.

### Решение и конкурентное преимущество TRIZ+RAG+LLM

AI-ассистент ведёт пользователя по этапам TRIZ (противоречие → матрица/приёмы → гипотезы), опираясь на RAG по корпусу методологии и корпоративной KB, а LLM — на генерацию и уточнение идей с обязательными цитатами источников. Преимущество перед «чатом с GPT»: воспроизводимый TRIZ-процесс, привязка к проверенным материалам заказчика и снижение галлюцинаций за счёт retrieval + структурированных промптов.

### Целевые пользователи

| Роль | Боль | Как помогает |
|------|------|--------------|
| Инженер-изобретатель | Не знает, какой приём выбрать; долгий поиск в литературе | Выявление противоречия, поиск по матрице и KB, гипотезы с цитатами |
| Руководитель R&D-проекта | Нет единого формата идей и истории решений | Сессии, оценка идей, экспорт для ревью |
| TRIZ-фасилитатор | Много рутины при воркшопах | Ускорение диалога, шаблоны этапов, KB под кейсы компании |
| Методолог / владелец KB | Устаревшие и разрозненные материалы | Управление KB, версии, качество retrieval |
| Интегратор (PLM/IDE) | Нужен API, а не только UI | REST API, мультиязычность, встраивание в процессы |

### Ожидаемые бизнес-результаты

| Метрика | Целевое направление (6 мес., MVP) |
|---------|----------------------------------|
| Time-to-first-hypothesis | −30–50% vs. ручной TRIZ-разбор |
| Доля гипотез с подтверждённой цитатой из KB | ≥ 80% |
| NPS / CSAT активных пользователей | ≥ 40 / ≥ 4.0 из 5 |
| Повторное использование (сессии/пользователь/мес.) | ≥ 3 |
| Доля идей, дошедших до оценки «go/explore» | +15% к baseline команды |

## 2. Epic

### Формулировка

**Дать R&D-команде воспроизводимый TRIZ-ассистент с RAG по методологии и корпоративной KB, от ввода задачи до оцениваемых гипотез с цитированием и сохранением сессий.**

### Бизнес-ценность

Ускорение изобретательского цикла, снижение зависимости от единичных TRIZ-экспертов, прозрачность решений для B2B-аудита.

### Техническая ценность

Единый pipeline: ingestion KB → Qdrant → LangChain/LlamaIndex → FastAPI → React; расширяемость API и мультиязычности.

### Definition of Done

| Критерий | DoD |
|----------|-----|
| Функциональность | 12+ user stories закрыты; сценарий «задача → противоречие → приёмы → гипотезы → оценка» E2E |
| RAG | Индексация KB; retrieval с цитатами; порог релевантности настроен |
| Качество | Приёмочные тесты по US; ручная валидация 10 кейсов TRIZ |
| Безопасность | Auth для B2B; изоляция KB по tenant (если применимо) |
| Ops | Деплой MVP (~10 users); логи, метрики latency/retrieval |
| Документация | API OpenAPI; гайд фасилитатору |

## 3. User Stories

**ID:** US-001  
**Роль:** Инженер-изобретатель  
**Цель:** Ввести описание технической задачи (текст/файл) и получить структурированное резюме для TRIZ-анализа.  
**Критерии приёмки:** Поля «система, полезная функция, вредный эффект»; лимит символов; валидация пустого ввода; сохранение черновика в сессии.  
**Приоритет:** Must  
**Зависимости:** US-011 (сессии)  
**Edge cases:** OCR/PDF с шумом; смешанный язык; слишком общая формулировка → запрос уточнения.  
**UX:** Мастер шага 1, примеры хороших формулировок, индикатор полноты.

---

**ID:** US-002  
**Роль:** Инженер-изобретатель  
**Цель:** Выявить и сформулировать техническое противоречие (улучшаемый / ухудшаемый параметр).  
**Критерии приёмки:** Предложение 1–3 вариантов противоречия; согласование пользователем; фиксация в сессии.  
**Приоритет:** Must  
**Зависимости:** US-001, US-007  
**Edge cases:** Несколько противоречий; пользователь отклоняет все варианты → ручной ввод.  
**UX:** Карточки «параметр A ↔ параметр B», подсветка из текста задачи.

---

**ID:** US-003  
**Роль:** Инженер-изобретатель

**Критерии приёмки:** Маппинг параметров на индексы матрицы; топ-N приёмов с номерами; объяснение связи с противоречием.

**Зависимости:** US-002  
**Edge cases:** Параметр вне матрицы → ближайший аналог + предупреждение; устаревшая версия матрицы в KB.

**ID:** US-004  
**Роль:** TRIZ-фасилитатор

**ID:** US-005  
**Роль:** Инженер-изобретатель  
**Цель:** Сгенерировать 3–7 проверяемых гипотез решения на основе приёмов и контекста задачи.

## 5. RAG-дизайн

### Стратегия чанкинга (TRIZ-корпус)

| Тип документа | Стратегия | `chunk_size` | `overlap` | Примечание |
|---------------|-----------|--------------|-----------|------------|
| Описание 40 приёмов | 1 приём = 1–2 чанка | 400–600 токенов | 80 токенов | Заголовок приёма в prefix каждого чанка |
| Матрица / таблицы параметров | Строка/ячейка + контекст | 150–250 токенов | 0 | Отдельная коллекция `triz_matrix`, не semantic-only |
| Кейсы / патентные обзоры | Recursive по абзацам | 512 токенов | 64 (12%) | Сохранять «Проблема → Решение → Эффект» в одном чанке где возможно |
| Стандарты / учебники | Иерархический (H2→H3) | 800 токенов max | 128 | Parent chunk (1200 tok) для контекста, child (512) для retrieval |
| Внутренние отчёты PDF | Layout-aware (Unstructured) | 512 | 64 | Таблицы → markdown-таблица отдельным чанком ≤ 300 tok |

**Правила качества:** минимальный чанк ≥ 100 токенов (иначе merge); максимум 1200 токенов; в каждый чанк — префикс метаданных: `[Приём №15 | Разделение]`; dedup по `content_hash`; язык определяется на ingestion.

### Гибридный поиск: BM25 + semantic

**Логика (MVP):**

1. **Query rewrite:** из сессии собирается запрос: задача + противоречие + выбранные параметры матрицы.
2. **Параллельный retrieval:**
   - **Dense:** Qdrant, top-50, фильтр `tenant_id`, `language`, `source_type`.
   - **Sparse:** BM25/pg `tsvector` по тем же фильтрам, top-50.
3. **Fusion:** Reciprocal Rank Fusion (RRF), `k=60`: `score(d) = Σ 1/(k + rank_i(d))` по обоим спискам.
4. **Rerank:** cross-encoder на top-30 fused → **top-8** в контекст LLM.
5. **Matrix path:** для US-003 отдельный детерминированный lookup (не RAG), RAG дополняет кейсами.

Порог отсечения: fused-кандидат с rerank score < 0.35 не попадает в prompt (→ «источник не найден»).

### Reranking

| Параметр | Значение MVP |
|----------|----------------|
| Модель | `BAAI/bge-reranker-v2-m3` (multilingual RU/EN) |
| Алгоритм | Cross-encoder: пары `(query, chunk_text)` → logits → softmax по батчу |
| Вход | top-30 после RRF |
| Выход | top-8, max 6 000 токенов суммарно в context window |
| Fallback | При недоступности reranker — только RRF top-10 |

### Снижение галлюцинаций (конкретные техники)

1. **Context-only generation:** system prompt — «утверждения о приёмах/кейсах только из CONTEXT».
2. **Citation-required:** JSON-schema ответа: каждая гипотеза содержит `citation_ids[]`; пустой массив → не показывать как факт.
3. **Retrieval gate:** если max rerank score < 0.35 — режим «уточняющих вопросов», без генерации приёмов.
4. **Groundedness check (post-LLM):** NLI-модель или LLM-judge: каждое предложение сопоставляется с chunk; несовпадение → `[не подтверждено]`.
5. **Temperature:** 0.2 для фактических блоков; 0.5 только для brainstorm-вариантов с маркировкой «идея».
6. **Matrix vs RAG:** номера приёмов из матрицы — только из справочника/БД, не из LLM.
7. **Chunk attribution prefix:** в prompt каждый chunk: `[ID: chunk_id | src: doc_title p.87]`.
8. **Feedback loop:** thumbs-down + причина → лог для дообучения промптов и blacklist чанков.

### Attribution (цитирование в ответе)

**Формат в UI и API:**

- В тексте: сноски **`[1]`**, **`[2]`** после утверждения.
- В блоке `sources[]` (параллельно ответу):

```json
{
  "citations": [
    {
      "ref": 1,
      "chunk_id": "kb_7f3a2c91_0042",
      "title": "TRIZ для инженеров",
      "locator": "стр. 87",
      "snippet": "Принцип динамизации предполагает...",
      "score": 0.82,
      "url": "/kb/documents/doc_triz_handbook_2024_v3#chunk_42"
    }
  ]
}
```

**Правила:**

- Один факт — одна сноска; агрегат нескольких чанков — `[1,3]`.
- Цитата дословна ≤ 200 символов; остальное — paraphrase с обязательной сноской.
- Приём из матрицы: `[M:15]` + ссылка на справочник приёма в KB.
- Если источник не из retrieval — только с меткой **`(экспертная интерпретация, без источника)`**.

### Онтология TRIZ (ключевые узлы и связи)

```
                    ┌─────────────┐
                    │   Problem   │  формулировка задачи, контекст системы
                    └──────┬──────┘
                           │ describes
                           ▼
                    ┌─────────────┐
                    │   System    │  надсистема, подсистемы, границы
                    └──────┬──────┘
                           │ has
                           ▼
                    ┌─────────────┐      improves          ┌──────────────────┐
                    │ UsefulFunc  │ ─────────────────────► │ EngParameter(A)  │
                    └─────────────┘                      └────────┬─────────┘
                           │ conflicts_with                      │
                           ▼                                     │ mapped_to
                    ┌─────────────┐                      ┌────────▼─────────┐
                    │ HarmfulEffect│                     │ MatrixCell       │
                    └──────┬──────┘                      │ (A × B → [pri])  │
                           │ worsens                       └────────┬─────────┘
                           ▼                                     │ suggests
                    ┌─────────────────┐                          ▼
                    │ EngParameter(B) │◄── worsens ────  ┌─────────────┐
                    └────────┬────────┘                  │  Principle  │ 1..40
                             │                           └──────┬──────┘
                             │ forms                            │ applied_in
                             ▼                                  ▼
                    ┌─────────────────┐                 ┌─────────────┐
                    │ TechContradiction│                │  Hypothesis  │
                    │  (A ↑ vs B ↑)   │                │  (solution)  │
                    └────────┬────────┘                 └──────┬──────┘
                             │ may_escalate_to                 │ supported_by
                             ▼                                 ▼
                    ┌─────────────────┐                 ┌─────────────┐
                    │ PhysContradiction│                │ Case / Chunk │ KB
                    │ (same param ±)  │                └─────────────┘
                    └─────────────────┘
```

**Связи для графа метаданных (теги при ingestion):**

| Связь | Пример |
|-------|--------|
| `Problem → hasContradiction → TechContradiction` | «объём↑ vs прочность↓» |
| `TechContradiction → mapsTo → MatrixCell` | cell(35, 28) → [15, 8, 35] |
| `Principle → illustratedBy → Case` | приём 15 + кейс из KB |
| `Hypothesis → usesPrinciple → Principle` | гипотеза ссылается на №15 |
| `EngParameter → synonymOf → Term` | «жёсткость» ↔ stiffness |

Онтология хранится как **JSON-LD / простой graph в Postgres** (`entities`, `relations`); чанки KB линкуются через `triz_principle_id`, `contradiction_params[]`.

### Стратегия нормализации и тегирования

**Нормализация (pipeline ingestion):**

1. **Кодировка и язык:** UTF-8; `langdetect` → `language: ru|en`.
2. **Термины:** маппинг на 39 параметров через глоссарий + fuzzy match (threshold 0.85); неизвестные → `unmapped_terms[]` для ручной привязки.
3. **Приёмы:** regex/lookup «приём 15», «динамизация» → canonical `triz_principle_id`.
4. **Единицы и числа:** NER для величин; в тексте — нормализация в SI где возможно (для кейсов).
5. **Дедуп:** `content_hash`; near-dup — cosine > 0.95 → merge metadata, один chunk в индексе.
6. **Версии:** `kb_version`, `supersedes_document_id`; reindex без удаления старой версии до TTL.

**Тегирование (обязательные и опциональные):**

| Тег | Обязательность | Источник |
|-----|----------------|----------|
| `source_type`, `tenant_id`, `language` | да | автомат |
| `triz_principle_id` | для приёмов/кейсов | правило + LLM-assist на ingestion |
| `contradiction_params[]` | для кейсов | LLM-assist + валидация по глоссарию 39 |
| `content_type` | да | `principle \| case \| handbook \| patent \| glossary` |
| `domain` | опц. | заказчик / классификатор |
| `access_level` | да | RBAC при upload |
| `quality_tier` | опц. | expert_reviewed \| draft |

**Контроль качества:** чанки без `triz_principle_id` и `content_type=principle` не индексируются в «methodology»-коллекцию; только в `raw_corpus`.

**Стадии state machine:** `intake → clarify → contradiction → retrieve → generate → evaluate`.

### Извлечение противоречия из текста задачи

**Двухпроходный pipeline (детерминизм + LLM):**

1. **Структурирование (LLM, schema):** из текста извлекаются: `system`, `useful_function`, `harmful_effect`, `constraints`, `ideal_result (IFR)`.
2. **Параметризация:** LLM предлагает 2–4 пары «улучшаемый / ухудшаемый» параметр; **маппинг на 39** — через embedding similarity к глоссарию параметров (top-3 кандидата на параметр).
3. **Формулировка противоречия:** шаблон: *«При улучшении [A] ухудшается [B], потому что [механизм из текста]»*; 1–3 варианта с confidence.
4. **Валидация:** если confidence < 0.7 или параметры не из топ-3 глоссария — переход в `clarify` (вопросы пользователю).
5. **Matrix lookup:** пара `(param_A_id, param_B_id)` → список приёмов (источник P0, не галлюцинация).
6. **RAG-дополнение:** кейсы с `contradiction_params` ∩ {A, B} для иллюстрации.

**Уточняющие триггеры:** нет вредного эффекта; A и B синонимы; противоречие физическое (один параметр ±) → ветка ВПР (post-MVP).

### Пример диалога (4+ хода)

**Ход 1 — задача (пользователь)**  
> Нужно уменьшить вес корпуса насоса на 20%, не снижая прочность при давлении 40 бар. Сейчас усиливаем стенки — вес растёт.

**Ход 2 — уточнение (ассистент)**  
> Зафиксировал систему: **корпус центробежного насоса**, полезная функция — **выдерживать 40 бар**, вредный эффект при улучшении — **рост массы**. Уточните: (1) допустимы ли композитные материалы или только металл? (2) что критичнее — масса всего корпуса или только зоны фланца?

**Ход 3 — TRIZ-анализ (ассистент, после ответа «только металл, важна масса всего корпуса»)**  
> **Техническое противоречие:** при увеличении **прочности (парам. 14)** ухудшается **объём/масса материала (парам. 35)** — типичная связка при однородном утолщении стенок [M: cell 14×35].  
> **Рекомендуемые приёмы из матрицы:** 15 Динамизация, 3 Местное качество, 1 Сегментация, 40 Композитные материалы *(40 отфильтрован — только металл)*.  
> **Из KB:** кейс «ребристый корпус — прочность при меньшем весе» [1]; фрагмент про «нагрузку на форму, а не на массу» [2].

**Ход 4 — решение / гипотезы (ассистент)**  
> **Гипотеза 1 (приём 3 — местное качество):** переменная толщина стенки — максимум у фланца и ребёр жёсткости, минимум в зонах низкого напряжения. Механизм: перераспределение напряжений без сплошного утолщения. Риск: сложность литья/мехобработки. *[1]*  
> **Гипотеза 2 (приём 15 — динамизация):** сменяемые вставки/кольца усиления только на период пусковых нагрузок. Механизм: в steady-state меньше материала. Риск: уплотнения, усталость. *[2]*  
> **Гипотеза 3 (приём 1 — сегментация):** корпус из секций с разной геометрией ребра, сборка сваркой. Механизм: оптимизация каждой зоны отдельно. Риск: концентраторы напряжений в швах.  
> Рекомендую оценить гипотезы 1–3 по шкале feasibility. Нужен расчёт FEA для выбранного варианта?

**Поведение между ходами:** после хода 2 ответ пользователя пишется в `session.clarifications[]`; stage → `contradiction`; matrix + RAG запускаются до генерации (ход 3–4); все утверждения о приёмах — с `[M:…]` или `[n]`.

| **SP-07** | Лицензии и объём P1 KB заказчика | Список документов + legal OK | 2 дн. (параллель PO) |

**Правило:** без закрытия SP-01, SP-02, SP-04 не фиксировать US-004 и US-002 выше 13 SP; без SP-05 — не обещать US-006 в S6.

### Сводка по фазам и velocity

```
Pre-MVP (S0)     : Spikes + P0 KB + skeleton     [~15 SP эквивалент]
MVP (S1–S9)      : Must Have ~83 SP              [velocity 8–9/спринт]
Post-MVP (S10+)  : Should + scale debt ~21+ SP
```

**Критический путь:** P0 KB → US-004 → US-002 → US-005 → US-006 → pilot (S9).

## 10. Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| **Галлюцинации** (приёмы, факты, кейсы не из KB) | Высокая | Критическое | Context-only prompt; matrix lookup из БД; citation-required schema; rerank gate; groundedness check; температура 0.2; метка «без источника» |
| **Слабый retrieval** (низкий recall, шум в top-k) | Средняя | Высокое | Hybrid BM25+semantic, RRF, reranker; тип-aware chunking; SP-01/02; eval-набор 20+ запросов; boost P0; feedback → blacklist чанков |
| **Качество извлечения противоречий** (неверные 39 параметров) | Средняя | Высокое | Двухпроходный pipeline; глоссарий + embedding top-3; confidence < 0.7 → clarify; валидация PO на 30 эталонах; ручной override |
| **Масштабирование** (latency, cost, Qdrant) | Средняя (к Scale) | Среднее→Высокое | MVP: single node, кэш сессий; лимит top-k и токенов; batch ingestion; Scale: кластер Qdrant, очереди, CDN, model routing |
| **Доверие пользователей** (не принимают ответы AI) | Средняя | Критическое | Прозрачные цитаты [n]/[M]; пилот с фасилитатором; режим «только retrieval»; audit log; оценка гипотез US-009 post-MVP |
| Низкое качество / объём KB заказчика | Средняя | Высокое | P0 без заказчика; шаблон кейсов; методолог KB в RACI; quality_tier |
| Лицензии на учебники и патенты | Средняя | Среднее | SP-07; только internal docs; обезличивание |
| Утечка данных / мультитенант | Низкая→Средняя | Критическое | `tenant_id` в фильтрах Qdrant; JWT; изоляция коллекций; no PII в логах промптов |
| Vendor lock-in OpenAI | Средняя | Среднее | Абстракция LLM/embedding; spike cost; план Azure/private на Scale |
| Scope creep (ВПР, ARIZ полный, fine-tune) | Высокая | Среднее | Явный out-of-scope MVP; stage machine только tech contradiction |
| Недооценка SP (RAG + US-010) | Средняя | Среднее | Spikes до commit; буфер 15% в S8; velocity по факту S2–S3 |
| Ключевой TRIZ-эксперт недоступен | Низкая | Высокое | 10 эталонных кейсов в backlog; backup reviewer |

## 11. KPI и метрики

### Продуктовые и операционные

| Метрика | Формула | Цель MVP (6 мес.) | Цель Scale |
|---------|---------|-------------------|------------|
| Time-to-first-hypothesis | `median(t_generate_first) − t_session_start` | ≤ 15 мин (−30% vs baseline) | ≤ 8 мин |
| Adoption | `MAU / licensed_users` | ≥ 80% (8/10) | ≥ 60% enterprise seats |
| Session depth | `sessions_with_full_pipeline / total_sessions` | ≥ 50% до этапа generate | ≥ 65% |
| Citation coverage | `claims_with_citation / total_factual_claims` | ≥ 80% | ≥ 90% |
| User satisfaction | CSAT после сессии (1–5) | ≥ 4.0 | ≥ 4.2; NPS ≥ 40 |
| Hypothesis throughput | `hypotheses_rated_go_explore / hypotheses_generated` | ≥ 25% | ≥ 35% |
| E2E latency (p95) | `p95(t_last_byte − t_request)` полный ответ | ≤ 15 с | ≤ 8 с |
| Cost per session | `(tokens_in+out)×price + infra) / sessions` | baseline + отчёт | −20% при том же качестве |
| Incident rate | `P1_incidents / month` | 0 P1 в pilot | SLA 99.5% uptime |
| KB freshness | `% docs indexed < 30 days` | ≥ 70% P1 | ≥ 90% |

### RAG-метрики

| Метрика | Формула | Цель MVP | Цель Scale |
|---------|---------|----------|------------|
| **MRR@k** | `mean(1/rank_i)` где `rank_i` — позиция первого релевантного чанка в top-k | ≥ 0.65 @k=8 | ≥ 0.75 |
| **NDCG@k** | `DCG@k / IDCG@k` по graded relevance (0–3) | ≥ 0.70 @k=8 | ≥ 0.80 |
| **Relevance** (retrieval) | доля чанков с expert label «релевант» в top-k | ≥ 75% @k=8 | ≥ 85% |
| **Faithfulness** | `supported_claims / total_claims` (NLI/judge vs chunks) | ≥ 85% | ≥ 92% |
| **Answer relevance** | LLM-judge: ответ решает запрос (1–5), mean | ≥ 4.0 | ≥ 4.3 |
| **Citation precision** | `correct_citation_links / total_citations` | ≥ 90% | ≥ 95% |
| **Hallucination rate** | `unsupported_factual_claims / total_factual_claims` | ≤ 10% | ≤ 5% |
| **Empty retrieval rate** | `queries_with_max_score < threshold / queries` | ≤ 15% | ≤ 8% |

**Измерение:** эталонный набор 30–50 пар (запрос, gold chunks) — обновление ежемесячно; faithfulness — на 20 сгенерированных ответах/релиз; дашборд Langfuse/Phoenix + продуктовые метрики в Postgres.

## 12. Итоговые рекомендации

### Топ-5 архитектурных решений

1. **Гибридный RAG (BM25 + dense + RRF + rerank)** — TRIZ-термины и номера приёмов плохо ловятся только embeddings; hybrid даёт стабильный recall для MVP и измеримый путь к Scale.
2. **Детерминированная матрица вне LLM** — приёмы по ячейке из JSON/SQLite, не из генерации; главный рычаг против галлюцинаций и роста доверия.
3. **Stage machine диалога** (`intake → clarify → contradiction → retrieve → generate`) — воспроизводимый TRIZ-процесс, проще тестировать и продавать B2B, чем свободный чат.
4. **Citation-required structured output** — гипотеза без `citation_ids` не в UI как факт; связывает продуктовый KPI с архитектурой.
5. **Postgres (сессии, артефакты) + Qdrant (chunks)** — разделение operational state и retrieval; проще auth, audit и reindex KB без миграции диалогов.

### Что НЕ строить в MVP

- Полный ARIZ / физические противоречия / ВПР-ветка как продукт
- Fine-tuning LLM и custom embedding training
- Multi-tenant SaaS с биллингом и self-service onboarding
- Real-time collaborative editing сессий
- Автоматический patent search / FTO
- Webhook/async API и внешние интеграции PLM (кроме минимального REST post-MVP)
- On-prem LLM и air-gapped deploy
- Автоматическая генерация KB из патентов без human review
- A/B мульти-модельный routing и LangGraph (достаточно LCEL chains)
- Продвинутая аналитика и BI-портал

### Порядок реализации

1. P0 KB: матрица, 40 приёмов, 39 параметров (JSON + ingestion).
2. Spikes: chunking, BM25, маппинг параметров (SP-01, 02, 04).
3. Skeleton: FastAPI, Postgres, Qdrant, auth, CI/CD.
4. US-008 сессии + US-001 ввод задачи.
5. US-007 диалог + stage machine.
6. US-002 противоречие + глоссарий.
7. US-003 матрица (lookup).
8. US-004 retrieval (dense → hybrid → rerank).
9. US-006 цитаты в API и UI.
10. US-005 гипотезы + groundedness gate.
11. US-010 KB admin + P1 корпус.
12. E2E QA, pilot 10 users, observability.
13. Post-MVP: US-011 API, US-012 i18n, US-009 оценка.

### Критические зависимости

```
P0 KB + лицензии
    → ingestion pipeline
        → hybrid retrieval (US-004)
            → contradiction mapping (US-002) + matrix (US-003)
                → hypotheses + citations (US-005, US-006)
                    → pilot и KPI faithfulness

TRIZ-эксперт (PO) → эталонные 30 кейсов → все RAG/KPI пороги

US-008 sessions → US-007, US-001 → весь E2E

SP-04 закрыт → фиксация оценки US-002

Заказчик P1 KB → ценность US-004/US-010; без P1 — только методология P0
```

**Главный принцип MVP:** сначала **доверие** (источники, матрица, цитаты), затем **скорость генерации** и расширение языка/API.
