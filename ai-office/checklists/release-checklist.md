# Release checklist

Объём проверки зависит от **типа релиза** (сервисы в Docker, только MQL5, только Pine, документация и т.д.). Отмечай **только применимое**; для узкого релиза остальное — N/A с пояснением.

## Перед релизом

- [ ] Все целевые Story/Task по релизу закрыты
- [ ] Код прошёл code review (Team Lead)
- [ ] Нет хардкода секретов, портов, URL
- [ ] `.env.example` актуален и описан в README **(если в релизе есть конфигурируемые сервисы или инструменты)**
- [ ] Миграции БД (если есть) проверены и задокументированы

## Сборка и окружение (если применимо)

- [ ] **Build:** успешная сборка тем способом, который зафиксирован для релиза (`docker compose build`, `npm run build`, и т.п. — **если есть контейнеры/фронт**).
- [ ] **Run:** успешный запуск (`docker compose up`, systemd, локальный venv — **по контракту релиза**); сервисы стартуют без необоснованных падений.
- [ ] **Health / smoke:** `GET /health` → 200 **(если есть HTTP-сервис)**; основной сценарий проверен вручную или автотестами **как в спеке**.

Если релиз **без** долгоживущих сервисов (только EA/индикатор, только Pine, только docs) — этот блок заполняется частично или помечается N/A.

## По типу доставки

### Python service / backend

- [ ] Тесты/линтеры по политике проекта (если используются)
- [ ] Зависимости без известных критичных уязвимостей (по возможности)

### Frontend

- [ ] Production-сборка и смок ключевого UI-пути (если фронт в релизе)

### MQL5

- [ ] Сборка артефактов для релиза; проверка по критериям из спеки (брокер, логирование сделок — если применимо)

### Pine Script

- [ ] Версия скрипта зафиксирована; поведение соответствует спеке публикации

### Docs / config only

- [ ] Согласованность изменений; отсутствие секретов в репо; обновление инструкций

## Тесты и качество (общее)

- [ ] Unit/интеграционные тесты для критичной логики проходят **(если в проекте есть тесты и они в scope релиза)**

## Документация

- [ ] README: как запустить, переменные окружения, примеры **(для того, что входит в релиз)**
- [ ] Архитектура/изменения отражены в docs при необходимости
- [ ] API: Swagger/OpenAPI доступен и актуален **(если есть HTTP API)**

## Деплой (сервер) — если применимо

- [ ] На сервере актуальны: код, `.env`, образы **(если деплой на сервер есть)**
- [ ] Сервисы запущены через docker compose **или** systemd **или** иной согласованный способ
- [ ] Логи пишутся, ошибки не проглатываются **(для сервисного релиза)**
- [ ] Firewall/порты: открыт только нужный **(если есть сеть)**
- [ ] При смене ключей — ротация секретов, проверка доступа

## После релиза

- [ ] Проверка основного сценария на production/staging **(если есть среда)**
- [ ] Мониторинг логов в течение N минут/часов **(для сервисов)**
- [ ] План отката известен (образ/тег, откат EA, откат конфига — **по типу релиза**)
