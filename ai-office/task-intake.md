# Единый шаблон постановки задачи

Все задачи формулируются в иерархии **EPIC → Story → Task**. Заполнять до начала разработки. Утверждает тимлид.

**Кому отдавать постановку:** только **Team Lead** (`ai-office/agents/team-lead/role.mdc`). Пользователь не ставит задачи программисту и тестировщику напрямую.

---

## 1. EPIC (крупная область / система)

| Поле | Описание |
|------|----------|
| **Название** | |
| **Цель системы** | Что создаётся на уровне продукта |
| **Бизнес-задача** | Какую проблему решаем |
| **Архитектурная роль** | collector / strategy engine / indicator service / execution service / risk service / dashboard-api / alert service / reporting |
| **Основные компоненты** | Список |
| **Стек** | Python, FastAPI, Redis, PostgreSQL, Docker, … |
| **Платформа реализации (обязательно)** | MQL5 / Pine / обе |
| **Критерий эквивалентности сигналов (обязательно для "обе")** | Как проверяем совпадение логики между MQL5 и Pine (условия входа/выхода, допуск по бару/цене, список тест-кейсов) |
| **Взаимодействие с другими сервисами** | Кто вызывает, кто вызывает нас, формат данных |

---

## 2. Story (функция внутри EPIC)

Формат: **As a** \<actor\> **I want** \<capability\> **So that** \<result\>.

| Поле | Описание |
|------|----------|
| **Название** | |
| **As a** | Роль/система (традер, стратегия, дашборд, …) |
| **I want** | Возможность (получать свечи, видеть позиции, …) |
| **So that** | Результат/ценность |
| **Описание** | Детали функции |
| **Входные данные** | |
| **Выходные данные** | |
| **Ограничения** | |
| **Зависимости** | Другие Story, сервисы, API |
| **Acceptance criteria** | Чеклист «сделано», когда всё выполнено |

---

## 3. Tasks (инженерные шаги)

Каждый Task — одна конкретная, измеримая работа.

| # | Task (формулировка) | Ожидаемый результат | Место в коде/сервисе | Критерий завершения |
|---|---------------------|---------------------|----------------------|---------------------|
| 1 | | | | |
| 2 | | | | |
| 3 | | | | |

**Требования к Task:**

- Техническая формулировка (не «сделать сервис», а «реализовать модуль RSI», «добавить endpoint GET /api/v1/…», «создать схему в PostgreSQL»).
- Понятно, где в репозитории/сервисе это делается.
- Чёткий критерий завершения (тесты проходят, endpoint возвращает 200, миграция применена и т.п.).
- Для торговых задач обязательно указать платформу реализации: **MQL5 / Pine / обе**.
- Если указано **обе**, обязательно добавить критерий эквивалентности сигналов (что считается совпадением и как проверяется).

## 4. Дополнительно (при необходимости)

| Раздел | Содержание |
|--------|------------|
| **Архитектура** | Схема компонентов, потоки данных |
| **Структура проекта** | Новые папки, модули, сервисы |
| **API-контракты** | Метод, путь, тело запроса/ответа, коды ошибок |
| **Формат данных** | JSON-схемы, типы, примеры |
| **Риски и ограничения** | Производительность, совместимость, миграции |

## Пример заполнения (коротко)

**EPIC:** Indicator Service — расчёт торговых индикаторов и выдача по API.

**Story:** RSI calculation  
As a *strategy engine* I want *to request RSI values* so that *strategies can generate signals*.

**Tasks:**

1. Реализовать модуль расчёта RSI на Python.
2. Создать кэш в Redis для результатов индикатора.
3. Добавить endpoint `GET /api/v1/indicator/rsi`.
4. Добавить health endpoint.

После заполнения и утверждения использовать при разработке и отмечать выполненные Task/Story.
